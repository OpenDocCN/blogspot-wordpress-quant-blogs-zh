# 对比特币高频交易的思考 | Tr8dr

> 来源：[`tr8dr.wordpress.com/2015/03/14/musings-on-hft-in-bitcoin/#0001-01-01`](https://tr8dr.wordpress.com/2015/03/14/musings-on-hft-in-bitcoin/#0001-01-01)

我在加利福尼亚的一个数据中心运行着 4 个比特币三级交易所数据流，运行顺利（这比东海岸更靠近亚洲交易所和 Coinbase）。处理这些交易所的不稳定性连接（由于使用了基于 REST 的 API 而偶尔返回 502/503 错误）需要一些错误处理和指数退避。

我考虑将 Bitstamp 和 Kraken 加入混合，尽管它们规模较小。Bitstamp 在安全漏洞之后似乎有所恢复，而 Kraken 因其欧元交易而独一无二。

**高频交易机会**

比特币交易及订单量与超高速移动的股票、外汇或国债市场相比还有很大差距。虽说如此，考虑到该市场微观结构的透明度更高，市场制造商和短期自营交易具有巨大的潜力。然而，需要注意的是，许多交易所的交易成本巨大（10 个基点或 20 个基点以上的往返成本），尽管 BTCChina 的交易成本出奇地低（费用在提现时收取）。

我刚开始收集主要交易所的高频交易数据，所以尽管我在非常小的样本上测试了几种信号，但希望收集更多的数据集进行回测和微调。这两种测试过的信号是围绕短期动量检测的，所以可以用它来跟随动量，或者在市场制造引擎中用作避险。在后者的情况下，如果检测到单边方向，将移除市场的一边报价，避免逆向选择。

**限制**

一些市场试图采取措施，以保持其市场对低频/非高频专业交易者的理智。今天查看 Kraken API 时，注意到了以下内容：

> 我们已经实施了保护措施，以防止滥用/DoS 攻击以及由快速下单和撤单引起的**订单簿操纵**……
> 
> 用户的计数器每几秒减少一次，但如果计数器超过 10，用户的 API 访问将被暂停 15 分钟。用户计数器减少的速度取决于各种因素，但最严格的限制是每 5 秒减少一次。

因此，Kraken 将允许偶尔的订单调整或放置，但平均每 5 秒只允许一次调整。对于市场制造商来说，每 5 秒（或 10 秒内两边）一次太受限了。如果市场有变动，这将成为市场制造商和趋势/动量跟随者的一个问题。或许这个限制应该根据价格变动进行调整，或者与更高的限制+最小生存时间（TTL）结合。

我实际上**支持**市场中的某些限制。尤其是股票市场，需要进行整治，而不是通过新的税收或考虑不周的监管（例如 reg-NMS），而是通过：

1.  基于时间的优先级/价格层次（信不信由你，有些订单类型允许高频交易跳过队列）

1.  最小的 TTL（生存时间）

1.  对某一证券设定合理的最大订单数/单位时间

第 2 点和第 3 点将消除许多不利于市场的交易所游戏。
